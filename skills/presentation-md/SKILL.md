# プレゼンモード用 Markdown 作成

md-editor のプレゼンモードは、**PowerPoint等を作らず、1枚の `.md` をそのまま16:9スライドに
見立てて発表する**機能。HTMLプレビューと同じレンダリング結果を、スライド単位に切り出して
パワポ風に見せる。このスキルは、その仕組みに最適化された Markdown を書くためのもの。

最も大事なのは次の2点。これさえ守れば破綻しない：

1. **スライドは `---`（区切り線）で区切る。`---` の前後には必ず空行を入れる。**
2. **各スライドは「見出し → 直後の1段落（キーメッセージ）→ 本文」の順で書く。**

詳しい挙動・エッジケースは [references/slide-syntax.md](references/slide-syntax.md) に、
そのまま開いて確認できる完成例は [assets/template.md](assets/template.md) にある。
迷ったらこの2つを読む。

## プレゼンモードのスライド構造（最重要）

プレゼンモードは生mdではなく**レンダリング後のHTMLを分割**する。各スライドは固定の3ゾーンに
割り当てられる：

| ゾーン | 何が入るか | 見た目 |
|---|---|---|
| **タイトル** | スライド先頭の見出し | 全ページ統一サイズで上部に固定表示 |
| **キーメッセージ** | 見出し直後の **最初の段落1つ** | 太字＋アクセント下線で強調（＝そのスライドの結論） |
| **本文** | それ以降すべて（箇条書き・表・図など） | 枠に収まるよう自動縮小。収まらなければ縮小下限でスクロール |

この構造から、各スライドの理想形はこうなる：

```markdown
## スライドのタイトル

ここに「このスライドで言いたい結論」を1文で書く（＝キーメッセージ）。

- 根拠・詳細1
- 根拠・詳細2
- 根拠・詳細3
```

見出し直後の段落が自動でキーメッセージ枠（アクセント下線）になる。ビジネス／コンサル資料の
「メッセージライン」がそのまま映える設計なので、**1スライド＝1メッセージ**を徹底する。

## スライドの区切り方

### 基本：`---` で明示的に区切る（推奨）

```markdown
## 1枚目

1枚目のメッセージ。

- 中身

---

## 2枚目

2枚目のメッセージ。

- 中身
```

**`---` の直前には必ず空行を置く。** 空行が無いと `テキスト\n---` が見出し（setext H2）と
解釈されてスライドが割れない。区切り線そのものはスライドに表示されず、境界としてだけ働く。

### 補足：見出しでも自動で割れる

`---` が無くても **H1 と H2 は自動的に新しいスライドの先頭になる**。
逆に言うと、**スライドの途中（本文）で H1・H2 を使うと、そこで意図せず割れる**。
スライド内で小見出しを使いたいときは **H3 以下（`###`）** を使うこと。

迷ったら「スライドの境目は `---`、スライド内の小見出しは `###`」と統一すると事故らない。

## 表紙（タイトルスライド）

**先頭の見出しを H1（`#`）にすると、そのスライドは表紙レイアウト**（縦中央・特大タイトル・
左揃え）になる。直後の段落はサブタイトル扱い（下線なし）。

```markdown
# 提案タイトル

サブタイトル / クライアント名 / 日付など

---

## アジェンダ
...
```

中身スライドの見出しは H2（`##`）に揃え、表紙だけ H1 にするのが基本形。

## プレゼンモードで使える主な記法

レンダリングは markdown-it ベース。以下はスライドできれいに出る：

- **箇条書き / 番号リスト** … 本文の主役。1スライド5〜7項目を目安に。
- **GFM表** … 自動で横スクロール対応。列が多すぎる表は避け、絞る。
- **太字・斜体・`インラインコード`・リンク**
- **コードブロック**（言語指定でシンタックスハイライト）… 長すぎると縮小されるので要点だけ。
- **Mermaid図** … ` ```mermaid ` フェンス。フロー・シーケンス・ガント等が図として描画される。
  ノードラベル内で改行したいときは `\n` ではなく **`<br>`** を使う（flowchartでは `\n` が
  そのまま文字として出てしまう）。
- **コールアウト** … `> [!NOTE]` `> [!TIP]` `> [!IMPORTANT]` `> [!WARNING]` `> [!CAUTION]`。
- **画像** … `![alt](path)`。読み込み後に高さに合わせて自動fitされる。
- **タスクリスト** … `- [ ]` / `- [x]`。

注意点：

- **単一改行が `<br>`（改行）になる**（breaks有効）。段落を分けたいときは空行を挟む。
- **数式（KaTeX）はプレゼン／HTML出力では確実でない**ため、重要な内容を数式だけに頼らない。
- 1枚に詰め込みすぎると本文が自動縮小され、限界を超えるとスライド内スクロールになって
  「発表で全部見えない」状態になる。**詰まってきたらスライドを分割**する。

## 作成ワークフロー

### A. トピックから新規生成するとき

1. **狙いを掴む**：聞き手・目的・結論・時間（≒枚数）をユーザーに軽く確認する（不明なら推定して進め、後で直せる前提で出す）。
2. **ストーリーラインを先に作る**：表紙 → アジェンダ →（課題 → 示唆 → 提案 → 効果 → 次アクション 等）→ まとめ、のように、各スライドの**キーメッセージを1行ずつ**並べて骨子を固める。コンサル資料は「説明 → 示唆（implication）」の順（CLAUDE.md 出力規約）。
3. **各スライドを3ゾーンで肉付け**：見出し＝論点、直後の1段落＝結論メッセージ、本文＝根拠（箇条書き／表／Mermaid図）。
4. **1スライド1メッセージ**と**詰め込みすぎない**を守る。
5. ファイルに保存して、プレゼンモードで開いて確認するよう案内する。

### B. 既存の文章・メモ・アウトラインを整形するとき

1. 元テキストを**論点ごとのブロック**に分ける。
2. 各ブロックに**1行のキーメッセージ（結論）**を立てる。元に結論が無ければ要約して作る。
3. ブロックを `---` で区切り、見出し＋メッセージ＋本文の3ゾーンに割り付ける。
4. 箇条書きが長い段落は要点に圧縮（スライドは読む物でなく見る物）。必要なら表やMermaid図へ。
5. **元の文章の意図・数値は変えない**。脚色や数値の創作はしない（不明な数値はプレースホルダにして注記）。

## 出力規約

- 言語は日本語をデフォルトとする（ユーザーの指示が英語なら英語）。CLAUDE.md の出力規約に従う。
- ファイル名は `YYYYMMDD_プロジェクト名_成果物名.md` を基本とする（例：`20260628_提案_SCM変革.md`）。
- 保存場所はユーザーに確認するか、指定が無ければ作業ディレクトリに置いて場所を伝える。
- 機密（社名・具体数値）はマスクして扱う。
- 完成後、**「md-editor で開き、表示メニューやツールバーからプレゼンモードに切り替えると確認できる」**ことを一言添える。

## 仕上げチェックリスト

書き終えたら以下を確認する：

- [ ] スライドは `---` で区切られ、**`---` の前に空行**がある
- [ ] 表紙が H1、中身スライドの見出しが H2 になっている
- [ ] スライド本文の中で H1・H2 を使っていない（小見出しは H3 以下）
- [ ] 各スライドの見出し直後に**1段落のキーメッセージ**がある（1スライド1メッセージ）
- [ ] 1枚あたりの本文が詰め込みすぎていない（目安：箇条書き5〜7行、長文はスライド分割）
- [ ] 表・コード・図が極端に大きくなっていない（縮小スクロール頼みになっていない）
