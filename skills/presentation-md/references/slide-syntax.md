# プレゼンモードのスライド分割・レイアウト仕様（詳細）

このファイルは、md-editor のプレゼンモードが Markdown をどう解釈してスライド化するかの
正確な挙動をまとめたもの。SKILL.md の要約で迷ったとき、エッジケースを確認したいときに読む。
（実装：`src/presentation.ts` / `src/styles/presentation.css` / `src/render-pipeline.ts`）

## 分割アルゴリズム（正確な挙動）

プレゼンモードは**生の Markdown を分割しない**。まず HTML プレビューと同じパイプラインで
レンダリングし、できた DOM の最上位要素を順に走査して、次の規則でスライドにグループ化する：

1. `<hr>`（Markdown の `---`）に当たったら、そこまでを1スライドとして確定し、`<hr>` 自身は
   捨てる（スライドには表示されない、純粋な境界）。
2. `<h1>` か `<h2>` に当たったら、それまでの内容を1スライドとして確定し、その見出しを
   **次スライドの先頭**にする。
3. それ以外の要素は現在のスライドに積む。

### ここから導かれる重要な帰結

- **`---` の前に空行が必要。** `テキスト` の直後の行に `---` があると、markdown-it はそれを
  setext H2（テキストを見出しにする下線）と解釈し、`<hr>` にならない。結果スライドが割れない。
  必ず空行を挟む：

  ```markdown
  本文の最後の行。

  ---

  ## 次のスライド
  ```

- **本文中で H1・H2 を使うと、そこでスライドが割れる。** スライド内に小見出しが欲しいときは
  `###`（H3）以下を使う。H3〜H6 は境界にならず、本文ゾーン内にそのまま表示される。

- `---` と見出しの両方があっても二重に空スライドは生じない（`<hr>` で確定 → 直後の見出しは
  空の現在グループには積まれず、次グループの先頭になるだけ）。なので
  「各スライドを `---` で区切り、それぞれ `##` で始める」書き方が最も安全。

## 3ゾーンへの割り付け

確定した各スライド（要素の並び）は、先頭から次の順で3ゾーンに割り当てられる：

1. 先頭要素が**見出し（H1〜H6）**なら → **タイトルゾーン**（無ければタイトル無しスライド）。
2. その次の要素が**段落 `<p>`** なら → **キーメッセージゾーン**（太字＋アクセント下線）。
3. 残り全部 → **本文ゾーン**。

注意：

- キーメッセージになるのは「見出し直後の最初の `<p>` ひとつだけ」。見出しの直後が箇条書きや
  表だと、キーメッセージゾーンは空になり全部が本文に入る（アクセント下線の強調は付かない）。
  → 結論を強調したいなら、**見出しの直後は必ず1段落の文**にする。
- 見出しが無いスライドでも、先頭が段落ならその段落がキーメッセージ扱いになる。意図しない
  強調を避けるため、中身スライドは基本「見出しで始める」。

## 表紙（タイトルスライド）レイアウト

スライド先頭の見出しが **H1（`#`）** のとき、そのスライドだけ表紙レイアウトになる：

- 紙面の**縦中央**に配置、左揃え、**特大タイトル**。
- 直後の段落は**サブタイトル**（太字・下線なしの控えめスタイル）。

中身スライド（H2始まり）は通常レイアウト（上部にタイトル固定＋下線付きメッセージ＋本文）。

## 本文の自動フィットとスクロール

- スライドは論理サイズ **1280×720（16:9固定）** のキャンバスに描画され、表示領域に合わせて
  全体が拡縮される（レターボックスあり）。
- 本文ゾーンは、枠に収まらない場合 `zoom` で自動縮小される。縮小率の下限は **0.55**。
- 下限（0.55）でも収まらないスライドは、**本文ゾーン内が縦スクロール**になる。
  → 発表中にスクロールしないと全部見えない状態になるので、**そうなったらスライドを分割**する。
- 目安：本文は箇条書き5〜7行、または中サイズの表1つ＋短いリード、または図1枚。これを大きく
  超えるなら2枚に割る。

## レンダリングで使える記法（プレゼン／HTML出力共通パイプライン）

`src/render-pipeline.ts` の markdown-it 構成に基づく。主なもの：

- **見出し** H1〜H6（H1/H2は前述のとおりスライド境界）。
- **箇条書き / 番号リスト**、ネスト可。
- **GFM表**：`.table-wrap` で囲まれ、はみ出す幅は横スクロール。
- **タスクリスト**：`- [ ]` / `- [x]` → 無効化チェックボックス。
- **コードブロック**：言語指定で highlight.js のシンタックスハイライト。
- **Mermaid**：` ```mermaid ` フェンスが SVG 図に変換される（flowchart, sequence, gantt, pie 等）。
  既定設定ではコード自体は隠れ、図だけ表示される。
  - ノードラベル内の改行は **`<br>`** を使う。`\n` は flowchart ではそのまま文字
    「\n」として表示されてしまうため使わない（例：`A[需要レビュー<br>営業] --> B[...]`）。
- **コールアウト（GitHub Alerts）**：`> [!NOTE]` / `[!TIP]` / `[!IMPORTANT]` / `[!WARNING]` /
  `[!CAUTION]`（文書設定でコールアウトが有効なとき。既定は有効）。
- **目次**：`[[toc]]` または `[toc]` を単独行に置くと目次が展開される（H1〜H3対象）。
  ただしプレゼンでは長い目次が1スライドに入りきらないことがあるので使用は控えめに。
- **太字・斜体・取り消し線・インラインコード・リンク・画像**。

### 方言・注意

- **`breaks: true`**：段落内の**単一改行が `<br>`（改行）**になる。意図せぬ改行を避けるには、
  続けたい文は同じ行に、段落を分けるときは空行を入れる。キーメッセージ段落の中で改行すると
  そのまま改行表示される（1枚に収めたい長文の折り返しには使える）。
- **生HTML は無効**（`html: false`）。`<div>` 等のHTMLタグはそのまま文字列として出る・効かない。
  レイアウトはあくまで Markdown の構造とプレゼンモードの3ゾーンに委ねる。
- **数式（KaTeX）はこの出力パイプラインでは保証されない**。スライドの要点を数式だけに依存しない。

## テーマ（配色・フォント）について

スライドの配色・フォント・見出しスタイルは、アプリの**文書テーマ設定**（設定モーダル）に
従う。アクセント色がキーメッセージの下線や見出しに反映される。**md 側でテーマは指定できない**
（front matter やHTMLでの色指定は効かない）ので、見た目の微調整はアプリ設定で行うよう案内する。
md 側でできるのは「構造を正しく作ること」だけ、と割り切る。
